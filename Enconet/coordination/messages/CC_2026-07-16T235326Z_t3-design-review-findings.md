FINDINGS:

- T3-F1 (medium) — the record→class map exists only in the Wiki-side design contract and is never published to the targets, so a published target is not clone-complete on record classes. `T3_GOVERNANCE_RECORD_CONTRACT.md:47` requires that "Every record declares or inherits one class", and `T3_TARGET_TEMPLATE_CONTRACT.md:50-60` does assign a class per asset — but that contract is explicitly a Wiki-side design source that is never rendered into a target ("target files must never import, link to, or read this Wiki workspace at runtime", `:6-8`). Of the nine published assets, only `event-log.template.md:3` declares its own class ("Append only."). `record-keeping.template.md:3-11` publishes the class *rules* but no mapping, so in the delivered repository nothing states that `support/current-status.md` is replaceable, that `support/log.md` is append-only, that `support/README.md` is controlled, or — most consequentially — that an **accepted ADR is immutable**. The immutability rule a maintainer would need is in the unpublished contract, not in the target. Fix (small, design-stage): add a record→class table to `record-keeping.template.md` (which already owns the class rules) mapping each published support path to its class, or declare the class in each template's header. This is the one thing standing between the design and its own clone-complete standard.

- T3-O1 (low, observation) — `support-index.template.md:26-27` links `../HANDOFF.md` and `../coordination/BOARD.md`. Both paths are correct and manifest-approved, but neither exists until the T4 (coordination) and T5 (handoff) slices land, while `PUBLICATION_ROLLBACK_MANIFESTS.md` sequences publication as "neutral skeleton, records/ADR, coordination, handoff, ...". At the skeleton commit the index therefore carries two dangling links, and `T3_TARGET_TEMPLATE_CONTRACT.md:89` requires validating relative links — so the T7.2 slice would either fail its own link check or teach reviewers to ignore it. Cheapest fixes: publish the index in the last neutral slice, or have the link validator accept manifest-approved paths that are not yet published within the same authorized publication. Worth settling in T6.1 before FIN publication rather than discovering it mid-slice.
